# 销售客户 Fail Path AI 助手

## 0. 结果定义

**解决问题**：帮助销售团队自动识别客户跟进过程中的风险信号（失联、停滞、低效跟进）减少人工盯表格的遗漏，以免错失挽回时机，提高客户转化率。
**成功的标准**：销售打开系统，能够录入客户信息与跟进记录，系统自动判定≥5种FailPath类型，调用大模型生成具体可操作的跟进建议，支持一键生成日报（TXT/Markdown/PDF），并能通过角色管理实现销售与主管的数据隔离和督促协作。一眼看到今天有哪些客户在"出问题"、问题是什么、下一步该怎么做，而不是自己翻表格猜。
**最关键的风险**：规则太松会漏掉真风险，太紧会产生大量误报，让销售对系统失去信任。因此，风险规则的阈值设置不当会导致误报或漏报；大模型API调用可能产生费用或延迟。

---

## 1. 功能概览

**客户状态管理**：录入客户信息与跟进记录，自动追踪阶段变化
**Fail Path 识别**：5 条风险规则自动检测，实时标记风险客户
**AI 跟进建议**：调用 glm-4-flash 大模型，为每个风险客户生成具体可操作的下一步建议
**每日日报**：一键生成风险客户汇总TXT或Markdown格式日报，支持下载
**Web 界面**：Streamlit 驱动的可视化操作界面
**多角色权限**：销售员只能管理自己的客户，主管可查看所有客户（只读）并发送督促消息

## 5. 消息推送设计

### 触发机制
**定时触发**：每日早上 9:00 自动生成日报并推送（工作日）
**事件触发**：当新客户被标记为风险时，立即推送告警通知

### 推送对象
**日报**：推送给销售主管（团队级概览）和对应销售（个人客户详情）
**风险告警**：推送给对应销售人员

### 内容组织
**日报**：
  概览区：今日风险客户数、各风险类型分布
  详情区：每个风险客户的阶段、风险类型、AI 建议
  行动区：按紧急程度排序的今日待办
**风险告警**：客户名 + 风险类型 + 建议摘要 + 快捷链接

### 推送渠道
1. **邮件**：通过 SMTP 发送 HTML 格式日报，附件含 PDF 版本
2. **企业微信/钉钉**：通过 Webhook 机器人推送 Markdown 消息卡片

### 技术实现思路
```
触发事件 → 日报生成器 → 渲染引擎（HTML/Markdown） → 分发器
                                                      ├→ SMTP 邮件发送
                                                      ├→ Webhook 推送
                                                      └→ 预留扩展接口
```
分发器采用策略模式，新增渠道只需实现 `send(content, recipients)` 接口。

## 6. 代码结构说明

### 模块划分

| 模块 | 文件 | 职责 |
|------|------|------|
| 入口 | `ui/app.py` | Streamlit界面，处理登录、页面路由、用户交互 |
| 数据层 | `db.py` | SQLite数据库连接、CRUD操作、事务管理 |
| 数据模型 | `models.py` | 定义Customer, FollowUp, RiskFlag, User, Reminder等数据类 |
| 规则引擎 | `risk_rules.py` | 5 条 Fail Path 识别规则，可配置阈值 |
| AI 建议 | `advisor.py` | 构建Prompt，调用大模型API（如glm-4-flash、DeepSeek等），返回具体建议 |
| 日报生成 | `report.py` | 生成TXT/Markdown日报 |
| 配置 | `config.yaml` | API Key、规则阈值、输出路径等 |

### 数据流转

```
用户录入 → db.py 存入 SQLite
                ↓
         risk_rules.py 扫描全部客户
                ↓
         匹配规则 → 写入 risk_flags 表
                ↓
         advisor.py 读取风险客户 → 调用 DeepSeek → 生成建议
                ↓
         report.py 聚合数据 → 生成 Markdown + PDF 日报
                ↓
         ui/app.py 展示结果 + 提供下载
```

### 关键设计决策
- **SQLite**：零配置，本地运行无需安装数据库服务
- **风险标记表独立**：risk_flags 与 customers 分离，支持历史追踪和状态变更（active → resolved）
- **规则可配置**：所有阈值通过 config.yaml 设置，无需改代码
- **AI 调用与业务解耦**：advisor.py 通过接口调用，可替换为其他模型

## 7. Agent 扩展设计

### 目标
将当前"人工触发"的工具升级为"自主运行"的 AI Agent，实现：定时自动检测、自动生成建议、自动推送日报。

### 架构设计

```
┌─────────────────────────────────────────────┐
│              Agent Controller               │
│  (调度中心：定时任务 + 事件驱动)             │
├─────────┬──────────┬──────────┬─────────────┤
│ Sensor  │ Planner  │ Executor │ Notifier    │
│ 感知器  │ 规划器   │ 执行器   │ 通知器      │
└────┬────┴────┬─────┴────┬─────┴──────┬──────┘
     │         │          │            │
  定时扫描   分析风险   调用API/    推送日报/
  CRM同步    制定计划   更新状态    发送告警
```

### 各模块职责

**1. Sensor（感知器）— 触发机制**
**定时触发**：Cron 任务，每日 9:00 执行全量扫描
**数据变更触发**：监听 CRM 系统的 Webhook，当客户状态更新时立即检查
**手动触发**：保留人工入口，销售可随时请求检查

**2. Planner（规划器）— 分析与决策**
运行全部风险规则，识别 Fail Path
对风险客户按严重程度排序
决定每个风险客户需要什么类型的跟进动作

**3. Executor（执行器）— 工具调用**
**Tool: 查询客户详情** → 从数据库/CRM 读取客户完整信息
**Tool: 生成跟进建议** → 调用 LLM 生成个性化建议
**Tool: 更新风险状态** → 将新发现的风险写入 risk_flags
**Tool: 生成日报** → 调用 report.py 生成报告文件
**Tool: 记录执行日志** → 记录 Agent 的每次操作

**4. Notifier（通知器）— 推送与反馈**
生成日报后自动推送
高风险客户发现后即时告警
推送渠道：邮件、企业微信、钉钉

### 与现有系统的边界

| 维度 | 当前工具 | Agent 扩展 |
|------|---------|-----------|
| 触发方式 | 人工打开网页点击 | 定时自动 + 事件驱动 |
| 数据输入 | 手动录入 | CRM API 自动同步 |
| 建议生成 | 手动点击"生成建议" | 自动生成并存储 |
| 日报输出 | 手动点击"生成日报" | 定时自动生成并推送 |
| 异常处理 | 人工判断 | Agent 自动识别并升级 |
| 人工介入 | 全程人工 | 仅在 Agent 不确定时请求人工确认 |

### 关键设计要点
1. **幂等性**：同一客户同一天不重复标记同一规则
2. **可中断**：Agent 运行中可随时暂停，下次从中断点继续
3. **可观测**：每次运行生成执行日志，可回溯 Agent 的决策过程
4. **安全边界**：涉及客户信息修改（如标记为"已流失"）需人工确认
